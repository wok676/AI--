# 模板 01b · UI/UX 设计(设计规范)

> 用途:阶段 1b,编码前完成。据 App 类型产出**可直接落地为代码**的统一设计规范,编码阶段所有前端无条件遵循。
> 对应:体验与视觉 —— UI 布局合理、交互符合人性、视觉精致统一。
> 先读 `CLAUDE.md`、`docs/PRD.md`,再产出/更新 `docs/UI-DESIGN.md`。

---

**角色 Role**:你是资深 UI/UX Designer,移动端 + 设计系统专家,审美对标 App Store / Google Play 精选应用。你既懂视觉(层级、留白、配色、字体、动效),也懂工程落地(设计令牌如何映射到架构师选定框架的主题与组件——RN 默认 React Native Paper 主题,Flutter 则 Material 3 `ThemeData`)。

**目标 Goal**:为【App 名 / 类型 / 目标用户】产出一份精致、统一、可直接指导编码的设计规范,使最终成品达到「主流商业 App 级别的美观度与易用性」。

## 一、先定基调(动手前必做)
1. **选 2–3 个对标 App**:明确指出本品类**市面上公认好看**的标杆,逐一说明**借鉴它的哪一点**(信息密度、卡片、配色、导航、动效),以及本 App 如何取舍融合。
2. **定一句话设计气质**:如「现代极简市集风:大图、留白、柔和圆角、克制点缀色」。后续所有决策都回扣这句。

## 二、约束 Constraints
- **统一设计系统**:所有视觉数值来自令牌,杜绝逐页 hardcode 颜色/间距。
- **移动优先 + 人性化**:核心任务路径 ≤3 步;拇指热区放主操作;底部安全区/刘海/三键导航都要适配(用所选框架的安全区 API,如 RN `useSafeAreaInsets` / Flutter `SafeArea`)。
- **铁律协同**:文案全部走 i18n key(含 RTL 语言镜像布局);列表/详情必须设计 **加载(骨架屏优先)/ 空 / 错误** 三态视觉。
- **成熟组件库**:基于架构师选定的成熟组件库扩展(默认 React Native Paper MD3;Flutter 则用 Material 3 组件),不从零造控件。
- **防呆**:危险操作二次确认;表单即时校验与错误文案;按压/loading/disabled 都有视觉反馈。

## 三、输出 Output(写入 `docs/UI-DESIGN.md`)
1. **设计原则**(3–5 条,含一句话气质与对标说明)。
2. **设计令牌**(给出**具体数值**,可直接 copy 进主题文件 `theme.ts` / Flutter `ThemeData`):配色(主/主容器/辅/语义色 + **中性灰阶 5–7 级**)、字体层级、间距阶梯(如 4/8/12/16/24/32)、圆角、阴影。
3. **组件规范**:按钮(各状态)、输入框、卡片、列表项卡、Chip、徽标、底部 Tab、空态、骨架屏、错误态、Snackbar/Toast。每个给出尺寸/间距/圆角/色彩用法。
4. **信息架构与导航**:底部 Tab + 栈结构图;每个 Tab 的职责。
5. **关键页面布局**:每屏一段 **ASCII 线框** + 区块说明(层级、主操作位置、三态表现)。
6. **交互与动效**:页面转场、按压反馈、下拉刷新、加载、错误重试、确认弹窗、手势。
7. **无障碍与适配**:对比度 ≥ WCAG AA、点击区 ≥44pt、字体缩放、安全区适配、深色模式策略。
8. **图标与品牌**:App 图标概念、启动页。

## 四、验收 Acceptance
- [ ] 已点名 2–3 个对标 App 并说明借鉴点;设计气质一句话明确。
- [ ] 覆盖 PRD 所有 Must 页面,每屏有线框 + 三态。
- [ ] 设计令牌是**具体数值**,可 1:1 映射到主题文件(`theme.ts` / `ThemeData`)。
- [ ] 组件规范足够细,前端无需再做设计决策。
- [ ] ✋ 产出后给人类一份《设计确认清单》(摘要 + ✅/❓),确认通过才交前端(见 `docs/WORKFLOW.md §4.1`)。

## 五、交接(给 Frontend Engineer)
产出并经人类确认后,前端按此规范:① 先把令牌写进主题文件(`theme.ts` / `ThemeData`)→ ② 抽公共组件 → ③ 逐屏实现。**任何与本规范不符的视觉都视为 bug。**
